# 车载语音助手 PRD（产品需求文档）

## 1. 产品概述

### 1.1 产品名称

CarAgent Voice — 仰望车载智能语音助手

### 1.2 产品定位

基于大语言模型的车载语音助手，支持自然语言控制车机功能（音乐、导航、空调等），并通过多渠道（飞书、QQ、微信、车机麦克风）提供统一的语音交互体验。

### 1.3 目标用户

- 仰望车主（通过车机语音交互）
- 车主及家庭成员（通过飞书/QQ/微信远程控制）

### 1.4 产品价值

- 免手操作：语音控制车机功能，驾驶更安全
- 远程控制：通过 IM 消息远程操作车机
- 开放架构：本地 LLM + 云端可选，数据可控

## 2. 功能需求

### 2.1 P0 — 核心功能

| ID | 功能 | 描述 | 验收标准 |
|----|------|------|----------|
| F01 | 语音唤醒 | 通过唤醒词激活语音助手 | 检测到唤醒词后 500ms 内进入录音状态 |
| F02 | 语音识别 (STT) | 将车机麦克风的语音转为文字 | 中文识别准确率 > 90%，延迟 < 2s |
| F03 | 意图理解 (LLM) | 大模型理解用户意图并生成执行计划 | 支持音乐、导航、空调等 10+ 指令 |
| F04 | 语音合成 (TTS) | 将回复文字转为语音播放 | 中文自然度可接受，延迟 < 1s |
| F05 | 车机控制 | 执行具体的车机操作 | 音乐播控（AUTOVOICE 广播）、导航搜索（高德 SDK）、A2UI 卡片渲染 |
| F06 | 飞书通道 | 通过飞书消息控制车机 | 文字消息 → 执行 → 文字回复 |

### 2.2 P1 — 增强功能

| ID | 功能 | 描述 |
|----|------|------|
| F07 | 连续对话 | 支持多轮语音交互，无需重复唤醒 |
| F08 | QQ 通道 | QQ 群/私聊消息控制车机，支持语音消息 |
| F09 | 微信通道 | 微信消息控制车机，支持语音消息 |
| F10 | 上下文记忆 | 记住当前对话上下文，支持追问 |

### 2.3 P2 — 未来功能

| ID | 功能 | 描述 |
|----|------|------|
| F11 | 多音区识别 | 区分主驾/副驾语音指令 |
| F12 | 情感识别 | 根据语气调整回复风格 |
| F14 | 多语言 | 支持中英文混合识别 |

### 2.4 已实现的 P2 → P1 功能

| ID | 功能 | 描述 | 验收标准 |
|----|------|------|----------|
| F13 | 主动关怀 | 基于车内场景(SCENE.md)主动导航+关怀乘员+生成信息卡片 | 选择场景卡片后自动导航并展示宝宝状态+关怀操作 |
| F15 | A2UI 信息卡片 | 结构化数据自动生成可视化卡片（天气/POI/状态/关怀） | 4种模板覆盖所有场景，卡片跨标签页保持，可删除不重复出现 |

## 3. 用户场景

### 3.1 场景一：车机语音控制音乐

```
用户："小迪小迪，播放周杰伦的青花瓷"
系统：[唤醒词检测] → [录音] → [STT: "播放周杰伦的青花瓷"]
      → [LLM 调 car-music skill, bash 工具]
      → [HTTP POST 127.0.0.1:18802/music/search {"song":"青花瓷","artist":"周杰伦"}]
      → [MusicController → BydMusicController.searchPlay]
      → [sendBroadcast(AUTOVOICE_COMMON_OPERATION, searchMusic) → BYD mediacenter 加载歌曲]
      → [sendBroadcast(AUTOVOICE_COMMON_CONTROL, play) → mediacenter 开始 PLAYING]
      → [TTS: "好的，正在播放周杰伦的青花瓷"]
      → [扬声器播放]
```

### 3.2 场景二：语音导航

```
用户："导航到大梅沙海滨公园"
系统：[唤醒词检测] → [录音] → [STT: "导航到大梅沙海滨公园"]
      → [LLM: intent=nav_search, dest="大梅沙海滨公园"]
      → [执行: car-cmd.sh nav_search dest=大梅沙海滨公园]
      → [执行: car-cmd.sh nav_select dest=大梅沙海滨公园 choice=1]
      → [TTS: "已为您导航到大梅沙海滨公园"]
      → [扬声器播放]
```

### 3.3 场景三：飞书远程控制

```
用户（飞书）："打开空调，温度调到24度"
系统：[飞书消息] → [OpenClaw] → [LLM]
      → [执行: am broadcast ... 或 UI 自动化]
      → [飞书回复: "已将空调温度调至24度"]
```

### 3.4 场景四：QQ 语音消息

```
用户（QQ群）：[发送语音消息 "今天天气怎么样"]
系统：[QQ bot 接收语音] → [转码 SILK→PCM] → [STT]
      → [LLM: 回复天气信息]
      → [TTS 生成语音] → [QQ 回复语音消息]
      → [同时文字回复]
```

### 3.5 场景五：主动关怀（宝宝打疫苗返程）

```
用户：在主动服务标签选择场景卡片"宝宝刚打完疫苗，返程路上"
系统：[ProactiveFragment] → 写入 SCENE.md（场景描述含宝宝哭闹、位置、乘员信息）
      → [OpenClaw Agent] 每轮读 SCENE.md
      → [car-nav] 检查导航状态 → 自动导航回家 (api /map/home)
      → [LLM] 生成温暖关怀回复 + 具体行动建议（摇篮曲、调温、安抚）
      → [A2UI 关怀卡片] 宝宝状态(哭闹不安) + 导航信息(目的地5.2km/12min) + 关怀操作(音乐/温度/安抚)
      → [GatewayClient] 检测 A2UI 协议 → [AGenUIFragment] 渲染卡片
      → [TTS] 语音播放关怀回复
```

## 4. 非功能需求

| 类别 | 指标 | 目标 |
|------|------|------|
| 延迟 | 端到端语音响应 | < 5s（唤醒词到播放开始） |
| 延迟 | 飞书消息响应 | < 10s |
| 可用性 | 系统启动 | 车机开机后自动启动，无需手动操作 |
| 可靠性 | 看门狗 | 进程崩溃后 30s 内自动重启 |
| 资源 | CPU 占用 | 待机 < 5%，活跃 < 50% |
| 资源 | 内存占用 | < 1GB（不含 LLM） |
| 并发 | 多 channel 同时请求 | 支持队列处理 |
| 安全 | API 认证 | Token 认证，LAN 限制 |

## 5. 技术约束

| 约束 | 说明 |
|------|------|
| 车机无 npm | Android shell 环境，无法安装 npm 包 |
| 车机无 curl/wget | HTTP 请求需通过 node-termux |
| input text 不支持中文 | Android input 命令限制，需用 AccessibilityService |
| 网络不稳定 | 车机 WiFi 可能断连，服务需自动重连 |
| LLM 在开发机 | MiniCPM 运行在开发机 172.20.10.5:8080 |
